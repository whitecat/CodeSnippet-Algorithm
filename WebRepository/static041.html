Question Why can't I declare a static method in an interface?
Derived from A question posed by stuart butler
Topics Java:Language, Java:Language:Interfaces, Java:Language:Syntax, Java:Language:Semantics
Author John Mitchell PREMIUM
Created Jun 27, 2000 Modified Jun 27, 2000


Answer
Because all methods declared in an interface are (implicitly) also "abstract" methods since, by definition, they do not define the (implementation of) the method.

That's sort of begging the question. The question could be rephrased, "I want to require that all classes implementing interface Foo define a static method Bar(). Why can't I do this by declaring a static method in interface Foo?" The answer is, because of subtle semantics involving the nature of what "static" really means. Perhaps another guru could be more specific.

OK, now that that's over with, I have another question:

Why can't I define a static method in an interface?

(Declare = spell out the message signature only; Define = provide implementation code too)

Answer: for the same reason you can't declare one: since it would cause problems with multiple inheritance. If object C implemented interfaces A and B, and both A and B defined a static method F, then some method invoked C.F(), then which F() would get invoked? The one from A() or the one from B()? We don't know which, and can't know, so Java doesn't let us complicate matters thus.

Actually, there is no ambiguity on this w.r.t. field declarations. That's because field declarations in Java interfaces are all, by definition, implicitly public static final -- i.e. those "declarations" are really just immutable constants. Because of that, all of the multiple paths of inheritance of those fields are collapsed/collapsible unambiguously.

Your restatement of the question has the same underlying cause. Interfaces are not even treated like the second class citizens that Java treats its classes. Interfaces in Java are basically just type constructs.

Re: That's sort of begging the question. The question could...
Darren Abbruzzese, Oct 10, 2001
The reason why you can't have a static method in an interface lies in the way Java resolves static references. Java will not bother looking for an instance of a class when attempting to execute a static method. This is because static methods are not instance dependent and hence can be executed straight from the class file. Given that all methods in an interface are abstract, the VM would have to look for a particular implementation of the interface in order to find the code behind the static method so that it could be executed. This then contradicts how static method resolution works and would introduce an inconsistency into the language.

Java doesn't have class methods, that why.
Ed Warnicke, May 18, 2006  [replies:1]

Not quite right. The reason is because in Java, static is really the same as "global". In general in OO languages there a three kinds of 'variables/methods':

  1. instance
  2. class
  3. global

Java doesn't really have class variables and methods. Many people think of 'static' as marking something as class associated, but it doesn't really. You can see this by trying the following: 

public class A {
    public static void printfoo() { System.out.println("A"); }

    public void doIt() { printfoo(); }

}

public class B extends A {
    public static void printfoo() { System.out.println("B"); }

}

If you have an instance b of class B, and call b.doIt(), it will print out "A". Why? Because printfoo isn't really a class associated method, it's a global, and it's reference in A.doIt() is a reference to that global.

So rephrase your question: 'I want to require that all classes implementing interface Foo define a global method Bar(). Why can't I do that by declaringa global method in interface Foo?". Do you see why that is sort of a silly thing? I agree with you it would be VERY useful to be able to require all classes that implement an interface to implement a class method Foo, but Java doesn't have class methods, so you can't.

Re: I don't get this answer
Alex Kinneer, Aug 10, 2006  [replies:2]

The fact that there is only one class C defined here is synthetic, it is completely irrelevant to the (correct) explanation regarding multiple inheritance. An interface is intended to provide an abstraction -- by definition you believe there are multiple implementations that might satisfy the abstraction or you would just be declaring it as a concrete class instead. So if you are using an interface, you would always expect at least the possibility of there being more than one implementation. So what is the implication?

First, a quick refresher on interfaces, since some in this thread seem to have forgotten how they really work. In normal source code using an interface, you would typically write something like the following:

SortingAlgorithm sorter = getSimpleSorter(); Collection sorted = sorter.sort(myCollection);

The abstraction here is that the interface (SortingAlgorithm) may be satistified by different implementations (say BubbleSort, QuickSort, etc...). The Java compiler encodes this as an INVOKEINTERFACE referencing SortingAlgorithm (*not* any concrete implementation).

Now, by comparison, a call to a static method is compiled as an INVOKESTATIC, which always references the explicit class defining the method. And the VM does *not* use virtual dispatch to resolve static methods. So suppose you have the following classes:

class Shape { }

class Rectangle extends Shape { static Rectangle newInstance() { .. } }

class Square extends Rectangle { static Square newInstance() { .. } }

Note that the covariant return types are perfectly legal, because Java does not view any of these methods as overriding. Consider the following:

Shape s = new Rectangle(); Rectangle r = s.newInstance();

The above is illegal, because the compiler cannot bind a valid matching function (this is actually the crux of the matter, it will be revisited in a moment).

Rectangle r = new Square(); Square s = r.newInstance();

The above is also illegal, because the call will bind to the definition in Rectangle, which returns the wrong type. This is because the compiler can easily determine the type of r. Note that if the methods were virtual, this would be legal (with a cast, of course), because the dynamic dispatch would act on the real receiver object, e.g. one of type Square.

As an aside, these examples demonstrate why it is advised never to call static methods on an object reference. Instead, you should always be explicit, as in:

Rectangle r = Rectangle.newInstance();

because this makes the semantics of what the compiler will do clearly match the source code.

So how does all this relate to an interface? Simple, because any method in an interface in Java has to use virtual dispatch by definition. This is the only sensible semantics to relate an abstraction to multiple concrete implementations, especially in a multiple interface inheritance situation.

Suppose two classes implement the same hypothetical static method from an interface, say for example, trusty old "foo()".

interface FooType { static int foo(); }

class FooImplOne { static int foo() { .. } }

class FooImplTwo { static int foo() { .. } }

Now suppose somewhere in your code, you want to invoke the method on a reference to the *interface*:

void someMethod(FooType ft) { int result = ft.foo(); }

What the compiler wants to do is resolve this to an INVOKESTATIC referencing FooType. But FooType doesn't actually contain any code! So what do to? It has to be resolved to a concrete class implementation. Setting aside the cost of 100% accurate static points-to analysis, the reference "ft" above could potentially reference objects of different types, so it is impossible to resolve this statically at compile time to an INVOKESTATIC in accordance with current Java semantics. You could have the VM search at runtime, but that's virtual dispatch, making it completely inconsistent with static semantics. So the only real solution is to push it back on the developer, by forcing a cast:

int result = ((FooImplOne) ft).foo();

But then you completely undermined the point of using an interface, because you assume that only one possible concrete implementation can reach "ft", or risk a ClassCastException!

I do entirely agree that there might be a potential usefulness to having a very semantically constrained version of static methods in interfaces though. Such a semantics would not permit a static method to be called on a reference to the interface (ever!) but would guarantee that implementors of the interface must define the method, for use in explicitly qualified static calls on the concrete type. An example would be encapsulated deserialization, where it would be nice for a class to know how to deserialize itself from a source and return an instance using a static method. e.g.:

InputStream file; MyClass mc = MyClass.deserialize(file);

I think it is additionally worth noting that the claim in this thread that Java does not have true class static members is completely false. The example given with the doIt() methods violates the principle of always qualifying a call to a static method with the desired type. The call to printfoo() is translated by the compiler to A.printfoo(), as a (dubious) convenience. But one could just as easily write B.printfoo() in the doIt() method of class A and get the implementation in class B. This is further underscored by the fact that B's printfoo() could be private, in which case B.printfoo() from A would be illegal. How can it be "global" if it can't be accessed from A? That's just nonsense, resulting from confusion caused by a lazy programming style. The fact that you didn't qualify a static method call with it's class type doesn't mean you should expect dynamic dispatch on the call!

Finally, to touch on the question of fields in interfaces. Problems with multiple inheritance are avoided because the compiler forces you to disambiguate conflicting field names. If a class inherits from more than one interface defining fields with the same name, you must always completely qualify the name of the field in the concrete class or the compiler will complain. A similar disambiguation would be required if static methods were permitted, to avoid concrete multiple inheritance conflicts.

The restriction is arbitrary
Dan Smith, Apr 11, 2007  [replies:2]

What a lot of misguided and confused discussion over a very simple question!

There is no good reason that interfaces must not allow static method declarations. It was an arbitrary choice made by the designers of the language.

A lot of the confusion here stems from the interpretation of "declare a static method." What I mean by it, and what I expect the poster meant by it, is to define a method tagged "static" and *not* (implicitly or explicitly) abstract -- naturally, the method would have a body.

Static methods behave just like static fields and static class declarations (both of which are allowed in interface declarations): their presense within a class or interface merely provides a namespace. They may be *shadowed* (JLS may use slightly different terminology), but not *overridden*. There is no dynamic dispatch on a static method. If two parent classes or interfaces define a field or nested class/interface with the same name, an unqualified reference to that name is considered ambiguous, and the same policy could be extended to static methods in interfaces.

Would it be useful to declare a static method inside an interface body? Sure, for the very same reason that it's useful to declare static fields and static classes there. It's just a convenient place to put things that would otherwise need a home. (For example, the methods in java.util.Collections could easily live in the java.util.Collection interface instead, eliminating the need for two class/interface declarations relating to the exact same thing.)
