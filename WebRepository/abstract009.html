In Java.nio.* there is a class called ByteBuffer

According to the documentation this is an abstract class.

From what I read in the tutorials this means that it can't be instantiated. Further, it means that the methods declared as ABSTRACT are just declarations and not implemented...

And yet in one of the examples I found that class in use!
So I tried it... and it works just fine. Not only can a make a ByteBuffer object, I can call on the abstract functions and they work as expected.

I can do this

 
 byte[] myBytes = new byte[100]
ByteBuffer myBuffer = ByteBuffer.wrap(myBytes);
short myShort = 0x0102;
myBuffer.putShort(myShort);
myBuffer.order(LITTLE_ENDIAN);
myBuffer.putShort(myShort);
 
for (i = 0; i<4; i++){
  system.out.println(myBytes[i]);
}


This will give output like
01
02
02
01

Which is just what I would expect of the order method, and the putShort method worked and were not abstract.
So these methods like putShort which are declared as abstract are actually implemented.
And I can make a ByteBuffer object such as myBuffer provided I use the static "wrap" function.

What I can't do is the following...

ByteBuffer newBuffer = new ByteBuffer(sizeOfBuffer);


That will yield an error claiming that it is not possible to instantiate an abstract class.

Okay but other than I don't have a normal constructor, what is the meaning of declaring a class abstract? The object called myBuffer seems to have all the properties of an instatiated object that I can tell except that maybe its underlying class, myBytes, is exposed. But the same would be true of DataBuffer and BufferedInputStream objects (whose underlying steams are passed in the constructor but are still exposed) and they are not declared abstract.

And what about the methods that are abstract? What is the difference between those an a normal method?

P.S. That is just pseudo code up there... I haven't tested it to compile or anything... it is just to give you an idea.

ByteBuffer myBuffer = ByteBuffer.wrap(myBytes);
This is NOT creating a ByteBuffer instance. It is creating an instance of something that extends ByteBuffer. You don't know what the actual class is, but it doesn't matter.

public interface IFoo
{
  public void bar(); // abstract by definition
}
 
public class MyFoo implements IFoo
{
  public void bar() { /* do something */ } // implemented
}
 
public class FooFactory
{
  public static IFoo giveMeAFoo() { return new MyFoo(); }
}
 
...
IFoo foo = FooFactory.giveMeAFoo();
foo.bar();

Your argument would also say the above is illegal because it looks like it is "creating" an IFoo. Of course you cannot actually instantiate IFoo because it is just an interface. But the code is creating something concrete which inherits from IFoo and only needs to deal with it at the interface level.

Read the source. ByteBuffer#wrap(...) returns an instance of java.nio.HeapByteBuffer, which is a package-private class. The reason for the public facing API specifying the return type as ByteBuffer is that this is an implementation detail that can be changed in future releases and you don't have to worry about that.

db

PS In case you're unaware of this, the source files are in the src.zip in your JDK directory. If you're using an IDE like NetBeans, you can open the zip file and peruse the sources in the IDE without having to unzip the file.

Or don't read the source: just observe that you aren't calling new ByteBuffer(), you are calling a static method whose return type is ByteBuffer, and whose internals you don't know anything about - but what you do know is that it can't call new ByteBuffer() either, for the same reason you can't. Ergo it must be supplying you with a concrete derived class.
