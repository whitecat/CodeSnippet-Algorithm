An abstract method requires implementation per instance. Static methods pertain to an overall class. A static method in an abstract class belongs to the abstract class, not potential implementations. It therefore doesn't make any sense to allow abstract static methods. Furthermore, static methods cannot be overridden, so again, abstract static methods would be an anomaly.

I don't think it's that simple. Smalltalk has what amounts to static methods with inheritance. Java can't do this because of more subtle (arguably poor) design decisions.

static method lookup in Java is determined at compile time, so can't adapt to subclasses that are loaded after code has been compiled in Java, but that's no reason not to have an abstract static that indicates that any sublcass of the class must also define a static implementation of that.

This would then be even more useful if it were syntactically easier to invoke a static method from a class object as in:

for (Class c : Class[] {Foo.class, Bar.class})
if (c.applies(x)) new C().process(x);
}

Also, since method lookup of static methods does look up the inheritance chain, an abstract qualifier would help ensure that callers don't mistakenly invoke the superclass version when they expect every class in some heirarchy to have it's own method implmentation. E.g. for c.applies() above, it would suck if someone forgot to put applies() on a subclass, so the superclass method was called implicitly.
